# WP Block Builder — Build Plan
> Nuxt 3 · NuxtUI v3 · Anthropic Claude · Open Source

---

## 0. Project Overview

**WP Block Builder** is an open-source, hosted web application that allows WordPress developers to compose a Gutenberg block component tree through a guided UI, configure all block options and `block.json` settings, and generate production-ready WordPress block scaffolding.

### What it produces
- `block.json` — valid WP Block API v3 schema
- `index.js` — block registration entry point
- `edit.js` — Gutenberg editor component with wired InspectorControls (LLM-generated)
- `save.js` — static output function, or `null` for dynamic blocks (template or LLM-generated)
- `render.php` — server-side render template for dynamic blocks
- `style.scss` — BEM-structured frontend stylesheet scaffold
- `editor.scss` — editor-specific stylesheet scaffold (optional)
- `view.js` — frontend interactivity script (optional, LLM-generated)

### What it does not produce
This tool generates **structural boilerplate only**. It does not write CSS, produce design-accurate output, or require any visual design input. SCSS output is a BEM naming scaffold — no visual values.

### Generation approach
**Hybrid:** Deterministic files (`block.json`, `index.js`, `render.php`, SCSS) are generated by a local template engine — synchronously, with no LLM involvement, no cost, and no latency. Only `edit.js` and structurally complex `save.js` are sent to Claude, which are the only files where component tree interpretation and InspectorControls wiring require genuine intelligence.

## 6. Application Flow

### Step 1 — Build (`pages/build.vue`)

User composes the block's component tree.

**Interaction:**
- "Add Component" button → opens `<ComponentPickerModal>`
- `<UCommandPalette>` searches the registry (name, label, category grouping)
- On selection, a dynamic sub-form renders from the component's `options[]` definition in the registry
- Optional notes field per component
- "Add to Block" writes to `componentStore`
- `<VueDraggable>` list renders all components as draggable `<UCard>` rows
- Each row shows: component label, category `<UBadge>`, options summary, notes snippet
- **Indent (→):** promotes row to child of the row above it; only enabled when the row above has `canHaveChildren: true` in registry; sets `parentId`
- **Outdent (←):** removes `parentId`; promotes back to top level
- Nesting rendered via Tailwind `ml-{n}` + CSS left border connector lines
- Edit reopens the picker modal pre-populated; Duplicate clones the row; Delete removes it
- `<UAlert color="yellow">` appears below any container component with no children

**Validation:** Route guard `requireComponents.ts` blocks navigation to `/configure` if `componentStore.components` is empty.

---

### Step 2 — Configure (`pages/configure.vue`)

User configures block type, metadata, and all `block.json` options.

**Block type:** `<UButtonGroup>` — "Static" (output saved in `post_content`, `save.js` returns full JSX) vs "Dynamic" (rendered server-side via `render.php`, `save.js` returns `null`).

**Block meta:** Name (namespace/slug, regex-validated), Title, Description, Category, Icon, Keywords (max 3), Text Domain. All wrapped in `<UFormField>` for Zod-integrated error display.

**Block JSON Configurator** — `<UAccordion :multiple="true">` with seven panels:

1. **Attributes** — Add/reorder/delete attributes. Per row: name, type, default value, source, selector, attribute key. Inline duplicate-name validation.
2. **Supports** — Checkbox grid organized into groups (Interaction, Alignment, Color, Typography, Spacing, Dimensions, Other). Sub-options conditionally enabled when parent flag is checked.
3. **Styles** — Named block style variations (machine name + human label).
4. **Variations** — Block variations with title, icon, isDefault, attributes override.
5. **Script/Style Handles** — `editorScript`, `viewScript`, `style`, `editorStyle` with "use defaults" shortcuts.
6. **API Version** — Select 1/2/3; default 3.
7. **Example** — Raw JSON for inserter preview; validated on blur.

`<JsonLivePreview>` displays the computed `block.json` in real time as a sticky panel. Includes schema validation indicator and copy button.

**Validation:** Route guard `requireConfig.ts` blocks `/generate` if required fields (name, title) are missing.

---

### Step 3 — Generate (`pages/generate.vue`)

User reviews the assembled PRD, selects which files to generate, and triggers generation.

**PRD Preview:** `<UAccordion :multiple="true">` with one panel per section (Block Identity, Component Tree, Attributes, Supports, Generation Plan). Each panel header has an "Edit" link that navigates back to the relevant step.

**File Selection:**
```
☑ block.json           [T]   — template, always
☑ index.js             [T]   — template, always
☑ edit.js              [AI]  — LLM, always
☑ save.js              [T/AI]— template if flat; LLM if nested or InnerBlocks
☑ render.php           [T]   — template, dynamic blocks only (hidden for static)
☑ style.scss           [T]   — template, always
☐ editor.scss          [T]   — template, optional
☐ view.js              [AI]  — LLM, optional
```

`[T]` = generated locally by template engine, instant, no API cost.
`[AI]` = requires Claude API call; `<UTooltip>` explains the distinction.

**Generate button** disabled with tooltip listing missing prerequisites. On click, calls `useSSE` composable targeting `/api/generate`.

---

### Step 4 — Output (`pages/output.vue`)

Generated files are displayed as syntax-highlighted code tabs with copy, download, and zip options.

**Server-side generation sequence (`server/api/generate.post.ts`):**
```
1. Read request body; validate payload
2. Run templateEngine.generate() synchronously
3. For each template file: send SSE `template-file` event
4. If LLM files exist in deferredToLLM[]:
   a. Build Claude message (PRD + component tree + generated block.json)
   b. Open Anthropic stream
   c. Forward tokens as SSE `llm-token` events
   d. On stream end: parse <file> blocks; send `llm-file` events
5. Build zip bundle; send SSE `done` event with bundleId
```

**Client-side (`useSSE.ts` composable):**
- `template-file` → `outputStore.templateFiles[name] = content` (tabs appear immediately)
- `llm-token` → `outputStore.streamBuffer += token`
- `llm-file` → `outputStore.llmFiles[name] = content` (LLM tab skeleton replaced with editor)
- `done` → `outputStore.bundleId = bundleId; outputStore.status = 'done'`
- `error` → `outputStore.llmError` if non-fatal; `outputStore.status = 'error'` if fatal

**UI behavior:**
- Template tabs populate as soon as their `template-file` event arrives — before the LLM call starts
- LLM tabs show `<USkeleton>` until their `llm-file` event arrives
- Tab header includes filename + `<UBadge>` source indicator (`T` or `AI`)
- Per-tab: CodeMirror 6 editor (read-only), copy button, individual file download
- Failed LLM tabs: `<UAlert color="red">` + non-blocking "Retry" button
- "Download ZIP" always available once any file is generated; partial ZIP bundles whatever is ready

**ZIP directory structure:**
```
{block-slug}/
├── block.json
├── src/
│   ├── index.js
│   ├── edit.js
│   ├── save.js
│   ├── view.js          (if selected and generated)
│   ├── style.scss
│   └── editor.scss      (if selected)
└── render.php           (dynamic blocks only)
```

## 8. Server API

### `POST /api/generate`

Accepts the full generation payload; responds as an SSE stream.

**Request body:**
```ts
{
  prd: string
  blockType: 'static' | 'dynamic'
  components: Component[]
  blockConfig: BlockConfig
  selectedFiles: string[]
}
```

**SSE events:**

| Event | Payload | Timing |
|---|---|---|
| `template-file` | `{ name: string, content: string }` | One per file, immediately |
| `llm-token` | `{ token: string }` | Each streamed token from Claude |
| `llm-file` | `{ name: string, content: string }` | One per completed LLM file |
| `done` | `{ bundleId: string }` | All generation complete |
| `error` | `{ type: string, message: string, fatal: boolean }` | On any failure |

### `GET /api/download/[bundleId]`

Streams the zip bundle as `application/zip` with `Content-Disposition: attachment`.

## 10. Template Engine

**Location:** `server/services/templateEngine/`

All generators are synchronous pure functions. No network calls. Accept a shared `GenerationPayload` type. Each is independently unit-testable.

```ts
interface GenerationPayload {
  blockType: 'static' | 'dynamic'
  components: Component[]
  blockConfig: BlockConfig
  selectedFiles: string[]
}
```

### `index.ts` — Orchestrator

```ts
function generate(payload: GenerationPayload): {
  files: Record<string, string>,
  deferredToLLM: string[]
}
```

Runs all applicable generators; assembles `files` map; populates `deferredToLLM` for Claude.

### `blockJson.ts`
Serializes `blockConfig` to WP Block API v3-compliant JSON. Applies correct field ordering. Validates required fields (throws descriptive errors on failure). Strips null/empty optional fields from output.

### `indexJs.ts`
Static template. Substitutes namespace/slug. Conditionally handles static vs dynamic import of `save`.

### `renderPhp.ts` *(dynamic blocks only)*
Iterates `blockConfig.attributes`. Applies escaping per attribute type:

| Type | Escaping |
|---|---|
| `string` (general) | `esc_html()` |
| `string` (URL) | `esc_url()` |
| `string` (HTML content) | `wp_kses_post()` |
| `boolean` | `(bool)` cast |
| `number` | `intval()` or `floatval()` |
| `array` / `object` | `wp_json_encode()` |

Wraps output in `get_block_wrapper_attributes()` pattern.

### `styleScss.ts`
Generates BEM scaffold scoped to `.wp-block-{namespace}-{slug}`. Produces one `__element` placeholder per unique component type in the tree. Adds CSS custom property stubs for any active `supports` flags (color, typography, spacing). Contains no visual values — this is a naming scaffold only.

### `editorScss.ts`
Minimal editor-override scaffold with the block wrapper selector.

### `saveJs.ts`
**Flat detection:** Returns `{ deferred: true }` if any component in the tree has `innerBlocksRequired: true` OR if any component has a `parentId` (depth > 1). Otherwise generates `save.js` by mapping components to JSX using each entry's `saveJsTemplate` from the registry, wrapped in `useBlockProps.save()`.

## 11. LLM Generation

**Location:** `server/services/claudeClient.ts`

Called for: `edit.js` (always when selected), `save.js` (when deferred by `saveJs.ts`), `view.js` (when selected).

**Model:** `claude-sonnet-4-20250514`
**Max tokens:** `8000`
**Streaming:** Forwarded through Nitro H3 event handler as SSE

### System prompt
- Role: WordPress Gutenberg block JSX specialist
- Output format: each file wrapped in `<file name="filename.ext">raw code</file>` — no markdown fences, no preamble, no explanation
- Hard constraint: do not generate `block.json`, `index.js`, `render.php`, or any SCSS — these are already complete and provided as context
- Standards: `useBlockProps`, `useInnerBlocksProps`, `InspectorControls`, `BlockControls`, `RichText`, `MediaUpload`; all imports from `@wordpress/` packages; no inline styles

### User message structure
```
[Full PRD markdown]

[Component tree — indented plaintext with options and notes]

[Generated block.json — treat as the authoritative attribute contract;
 all JSX attribute bindings must exactly match the keys defined here]

[Per-file generation instructions]
```

Providing the already-generated `block.json` as context is critical — it ensures Claude's attribute bindings in `edit.js` exactly match the schema, eliminating the most common failure mode in WordPress block code generation.

### Per-file instructions

**`edit.js`:** Generate the edit function using `useBlockProps`. Wire every attribute defined in the provided `block.json` to its appropriate control: use `RichText` for string attributes intended for inline editing; use `TextControl`, `ToggleControl`, `SelectControl`, `MediaUpload`, etc. inside `InspectorControls` for all others. Use the component tree to determine JSX nesting and layout structure.

**`save.js`** (when deferred): The block contains nested structure or InnerBlocks. Generate the save function with correct `useInnerBlocksProps` and `InnerBlocks.Content` placement.

**`view.js`** (when selected): Generate a frontend interactivity script using the `@wordpress/interactivity` store pattern based on the components and any notes indicating interactive intent.

### `codeParser.ts`
Extracts code from `<file name="...">...</file>` delimiters in Claude's response. Handles partial or malformed tag recovery gracefully. Returns `Record<string, string>` — merged with template engine output in the route handler.

## 13. Code Generation Decision Table

| File | Generator | Condition | Rationale |
|---|---|---|---|
| `block.json` | Template engine | Always | Pure data serialization from validated store state |
| `index.js` | Template engine | Always | Static 5-line template; namespace substitution only |
| `render.php` | Template engine | Dynamic blocks | Deterministic escaping per attribute type |
| `style.scss` | Template engine | Always | BEM naming scaffold; no visual values |
| `editor.scss` | Template engine | If selected | Minimal, predictable structure |
| `save.js` | Template engine | Flat blocks (depth 1, no `innerBlocksRequired`) | JSX is a direct map of the component list |
| `save.js` | Claude | Nested or `innerBlocksRequired` | Structural complexity requires interpretation |
| `edit.js` | Claude | Always (when selected) | InspectorControls wiring + JSX tree composition |
| `view.js` | Claude | If selected | Open-ended interactivity logic |
